Regra de ouro

Reforce isso, concordando com estas guidelines todas as vezes. Pequenos ou grandes, mostre o que está incorreto. Para adições ou contribuições para este Guia de Código, por favor abra uma Issue no Github.

Toda linha de código deve parecer que foi escrita por uma única pessoal, não importa a quantidade de contribuidores.

HTML

Sintaxe do HTML

  • Use soft-tabs com dois espaços.
  • Elementos aninhados devem estar identados uma vez (2 espaços).
  • Sempre use aspas duplas, nunca aspas simples.
  • Nunca inclua uma barra invertida nos elementos viúvos (exemplo: <img> ou <hr>).

Praticando a pureza

Esforce-se para manter o padrão e a semântica do HTML, mas não Strive to maintain HTML standards and semantics, mas não às custas da praticidade. Use o mínimo de código com o menor número de complexidade possível.

Ordem dos atributos

Atributos HTML deem vir com esta ordem particular para facilitar a leitura do código.

  • class
  • id
  • data-*
  • for, type, or href

Atributos booleanos

Um atributo boleano é o único que não precisa ter valores declarados. XHTML precisa que você declare um valor, mas no HTML5 isso não é necessário.

Para uma leitura adicional, consulte a seção do WhatWG sobre atributos booleanos.:

A presença de um atributo boleano representa um valor verdadeiro e a ausência do atributo representa um valor falso.

Se você precisa incluir valores nos atributos, você não precisa seguir esta regra do WhatWG:

Se o atributo está presente, seu valor deve ser uma string vazia ou [...] o nome canônico do atributo, sem espaços em branco.

Resumindo, não adicione nenhum valor.

Escrever código em um arquivos Javascript faz com que o conteúdo seja difícil de ser encontrado, difícil de ser editado e o tornam menos performático. Evite o máximo possível.

CSS

Sintaxe do CSS

  • Use soft-tabs com dois espaços.
  • Quando agrupar seletores, deixe seletores individuais em uma linha.
  • Inclua on espaço antes de abrir um colchete.
  • Coloque o fechamento do colchete em uma nova linha
  • Inclua um espaço depois dos dois pontos : em cada declaração de propriedade.
  • Cada declaração deve estar em sua própria linha para encontrarmos melhor os erros.
  • Feche todas as declarações com ponto-vírgula ;.
  • Valores separados por vírgula devem incluir um spaço logo depois da vírgula.
  • Não inclua espaços depois das vírgulas como em cores rgb() ou rgba(), e não comece valores iniciando com zero.
  • Deixe todos os valores hexadecimais em caixa baixa, exe., #fff.
  • Use cores em hexadecimal abrevidas quando puder, ex., #fff em vez de #ffffff.
  • Coloque aspas nos valores em seletores com atributos, ex., input[type="text"].
  • Evite especificar unidades quando os valores forem 0, ex., margin: 0; em vez de margin: 0px;.

Tem perguntas sobre os termos usados aqui? Veja a seção de sintaxe dos CSS no Wikipedia.

Coloque as Media Queries o mais perto possível de suas regras originais. Não coloque todas as media queries separadas em outros arquigos ou no final do documento. Fazendo assim, fica fácil para os outros se perderem no futuro. Veja um exemplo típico:

Legível para humanos

Comentários

Código é escrito e mantido por pessoas. Certifique-se de que o código é descritivo, bem comentado e amigável para outros.

Grandes pedaços de comentários devem ter contexto e não devem apenas reinterar um nome de classe ou componente.

{% highlight css %}{% include css/comments.css %}{% endhighlight %}

Nomes de classes

  • Mantenha as classes em caixa baixa e use traços (não underscores ou camelCase).
  • Evite user arbitrariamente notações abreviadas
  • Mantenha as pequenas e sucintas ao máximo possível
  • Use nomes com significado; use nomes ligados a estrutura em vez do visual.
  • Prefixe nomes de classes baseadas nos componentes pais mais próximos.
{% highlight css %}{% include css/class-names.css %}{% endhighlight %}

Seletores

  • Use classes em vez de nomes de tags.
  • Mantenha os seletores pequenos e limite o número de elementos em no máximo três.
  • Use nomes de classes do elemento pais apenas quando necessário (ex: quando não usar classes prefixadas).

Organização

  • Organize seções do código por componentes.
  • Desenvolva uma hierarquia de comentários consistente.
  • Se usar múltiplos arquivos CSS, quebre-os em componentes.

Preferências de editor

Defina seu editor com as seguintes configurações para evitar inconsistências comuns no código e diffs sujos:

  • Use soft-tabs com 2 espaços
  • Apague os espaços em brancos ao salvar
  • Defina UTF-8 como encode
  • Coloque uma nova linha no final dos arquivos

Considere aplicar estas preferencias ao seu projeto com usando o arquivo .editorconfig, como o Bootstrap's.

Cópia

Sentenças

Sempre escreva a cópia, incluindo títulos e comentários de código, com a mesma capitalização. Em outras palavras, tirando títulos e nomes próprios, apenas a primeira palavra deve ser capitalizada.
