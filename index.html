HTML

Sintassi

  • Non scrivere i tag in maiuscolo, compreso il doctype
  • Usa tab rimpiazzati da due spazi - è l'unico modo per garantire che il codice appaia nello stesso modo in ogni ambiente
  • Gli elementi annidati dovrebbe essere indentati una sola volta (con due spazi).
  • Usa sempre le doppie virgolette per gli attrbuti, mai le virgolette singole
  • Non mettere lo slash per chiudere i tag self-closing - la specifica HTML5 dice che sono opzionali.
  • Non omettere i tag di chiusura (es. </li> or </body>)

Praticità, non formalismo

Cerca di mantenere degli standard e una certa semantica, ma solo se questo non peggiora la praticità. Cerca di scrivere meno codice possibile e con meno annidamenti possibile.

Ordine degli attributi

Per facilità di lettura, gli attributi HTML dovrebbero seguire quest'ordine

  • class
  • id, name
  • data-*
  • src, for, type, href, value
  • title, alt
  • role, aria-*

Le classi permettono di riusare i componenti, quindi si scrivono per primi. Gli ID sono più specifici e dovrebbero essere usati il meno possibile (es.: per segnalibro interni ad una pagina), quindi si scrivono dopo.

Attributi booleani

Un attributo booleano non ha bisogno di un valore dichiarato. Lo standard XHTML ti obbliga a dichiarare un valore, ma per l'HTML5 è opzionale.

Per saperne di più, visita la sezione WhatWG riguardo gli attributi booleani:

La presenza di un attributo booleano su un elemento rappresenta il valore "vero" e l'assenza dell'attributo rappresenta il valore "falso".

Se devi per forza dare un valore all'attributo, anche se è puramente opzionale, segui questa linea guida del WhatWG:

Se l'attributo è presente, il suo valore deve essere o una stringa vuota oppure [...] il nome canonico dell'attributo, senza spazi bianchi né prima né dopo.

In breve, non aggiungere un valore.

CSS

Sintassi CSS

  • Usa tab rimpiazzati da due spazi - sono l'unico modo di assicurarsi che il codice sia rappresentato allo stesso modo in ogni ambiente
  • Quando raggruppi i selettori, ognuno deve avere la sua riga
  • Aggiungi uno spazio prima di aprire le parentesi graffe dei blocchi di dichiarazione, per leggibilità
  • Le parentesi graffe di chiusura devono avere la loro riga
  • Aggiungi uno spazio dopo : per ogni dichiarazione
  • Ogni dichirazione deve avere la sua riga per agevolare il report degli errori.
  • Concludi ogni dichiarazione con un punto e virgola. Per l'ultima dichiarazione del blocco è opzionale, ma il tuo codice è più fragile se non lo fai.
  • I valori delle proprietà che separati da virgole dovrebbero avere uno spazio dopo ogni virgola (es., box-shadow).
  • Non aggiungere spazi dopo le virgole all'interno di rgb(), rgba(), hsl(), hsla() e rect(). Questo aiuta a differenziare gli argomenti delle funzioni (con virgole, senza spazi) dalle liste di valori di proprietà (con virgole e spazi).
  • Non aggiungere zeri a sinistra dei valori delle proprietà o dei parametri dei colori, (es. .5 al posto di 0.5 e -.5px al posto di -0.5px).
  • Scrivi i colori esadecimali in minuscolo, es.: #fff. Le lettere minuscole sono più facili da riconoscere mentre scorri il documento perché hanno forme più peculiari.
  • Usa valori esadecimali abbreviati quando possibile, es.: #fff al posto di #ffffff.
  • Metti tra virgolette i valori degli attributi nei selettori, es.: input[type="text"]. Sono opzionali solo in alcuni casi ma è comunque buona norma metterle.
  • Evita di specificare le unità di misura per valori 0, es.: margin: 0; al posto di margin: 0px;.

Hai qualche domanda sui termini usati qui? Visita la sezione di sintassi del CSS su Wikipedia.

Ordine delle dichiarazioni

Le dichiarazioni di proprietà simili vanno raggruppate insieme secondo quest'ordine:

  1. Posizionamento
  2. Box model
  3. Tipografia
  4. Colori e forme
  5. Altro

Il posizionamento è il primo gruppo perché può rimuovere gli elementi dal flusso normale del documento e può sovrascivere lo stile del box model corrente. Il box model viene subito dopo perché dichiara le dimensioni e il posizionamento locale del component.

Tutto il resto si trove all'interno del component o comunque non influenza i primi due gruppi, per questo vengono messi alla fine.

Non usare @import

Rispetto ai <link>, gli @import sono più lenti, aggiungono inutili chiamate HTTP e possono provocare problemi imprevisti. Evitali se possibile e scegli una di queste alternative:

  • Usa elementi <link> multipli
  • Compila il tuo CSS con un preprocessore come Sass oppure Less così da produrre un singolo file.
  • Concatena i tuoi file CSS tramite alcune features disponibili in Rails, Jekyll ed altri ambienti.

Dichiarazioni singole

Nel caso in cui un blocco di dichiarazioni abbia una sola dichiarazione, prova a rimuovere gli a capo per una maggiore leggibilità e facilità di modifica. Ogni blocco di dichiarazioni che ha più di una dichiarazione dovrebbe avere una dichiarazione per linea.

La cosa più importante è l'individuazione degli errori - es., un validatore CSS dice che hai un errore di sintassi alla Linea 183. Se hai una dichiarazione per linea, sei sicuro di quale regola vada in errore.

Proprietà abbreviate

Limita l'uso delle proprietà abbreviate solo nel caso in cui tu debba impostare tutti i valori. Le proprietà abbreviate più comuni sono:

  • padding
  • margin
  • font
  • background
  • border
  • border-radius

Di solito non c'è bisogno di settare tutti i valori rappresentati nelle proprietà abbreviate. Per esempio, le intestazioni HTML hanno soltanto il margin superiore ed inferiore impostato di default, quindi quando necessario è bene sovrascrivere soltanto quei valori. Un valore `0` implica di sovrascrivere sia il default del browser o qualsiasi altro valore già dichiarato.

L'uso eccessivo delle proprietà abbreviate porta ad avere un codice più fragile con sovrascritture non necessarie ed effetti imprevisti.

La Mozilla Developer Network ha un ottimo articolo riguardo le proprietà abbreviate for those unfamiliar with notation and behavior.

Commenti

Il codice è scritto e manutenuto da persone. Assicurati che il tuo codice sia esplicativo, ben commentato e esplorabile dagli altri. I migliori commenti al codice forniscono un contesto oppure uno scopo. Non ripetere semplicemente il nome di un componente o di una classe.

Nomi delle classi

  • I nomi delle classi devono essere esclusivamente in minuscolo (niente trattini bassi o camelCase) e contenere al massimo lettere e trattini, che fungono da separatori naturali per classi simili (es.: .btn e .btn-danger).
  • Evita l'uso eccessivo delle abbreviazioni. Per esempio .btn va bene per indicare un bottone, ma .s non significa nulla.
  • Scegli nomi più brevi possibili
  • Scegli nomi descrittivi: preferisci nomi composti se possibile.
  • Aggiungi prefissi alle classi in base al padre più vicino.
  • Usa classi .js-* per indicare i bersagli degli script, ma cerca di non inserire questi classi nei file CSS

È possibile utilizzare queste regole anche per i nomi delle variabili Sass e Less.

Selettori

  • Usa le classi al posto dei tag generici per migliorare le prestazioni.
  • Evita di usare troppi selettori di attrbuto (es.: [class^"..."] per componenti usati spesso) perché le prestazioni del browser ne risentono significativamente
  • Scegli selettori brevi e cerca di limitare il numero degli elementi all'interno del selettore a tre.
  • Annida le classi (con un prefisso) rispetto al progenitore più vicino (es.: .genitore .figlio) solo quando necessario (es.: quando non usi le classe con prefisso).

Organizzazione

  • Raggruppa il codice per componente
  • Imposta una gerarchia consistente di commenti
  • Usa gli spazi bianchi con criterio in modo da separare le varie sezioni in documenti di grandi dimensioni
  • Quando utilizzi più file CSS, dividili per componente e non per pagina. Le pagine possono essere riorganizzate e i componenti spostati, mentre i componenti non dovrebbero cambiare

Impostazioni dell'editor

Aggiungi nel tuo editor le seguenti impostazioni per evitare errori comuni nel codice e modifiche selvagge:

  • Converti i tab in spazi bianchi.
  • Elimina gli spazi bianchi in eccesso quando salvi.
  • Imposta la codifica caratteri a UTF-8.
  • Aggiungi una riga vuota alla fine di ogni file.

Valuta se salvare queste impostazioni nel file .editorconfig del tuo progetto. Per esempio, guarda quello di Bootstrap. Scopri di più sul progetto EditorConfig.
